1. What Is This App?

This is our internal tool for tracking everything related to the Community-Based Sensor Network. It replaces the old Salesforce workflow and puts everything in one place — sensors, the communities they're deployed in, contacts at each site, service tickets, collocations, and audits.

Sensor Management

Track every sensor's location, status, install date, and full event history. Move sensors between communities with automatic logging.

Service Tickets

Track repairs and calibrations with QuantAQ. Log each ticket from opening through shipping, repair, progress notes, and return.

Collocations and Audits

Manage collocations and audits. Track progress, upload data, run regression analysis, and generate formatted reports.

Community Tracking

Keep tabs on each community's sensors, contacts, communications, and files. Organized with parent/child community relationships. Optional tagging for sub-networks such as Interior Wildfire Network or Muni of Anchorage Network.

2. Getting Started

To access the app, go to amqa-tools.github.io/sensor-network-tracker and sign up or sign in. To maintain site security, the email associated with your account needs to be added to the approved user list first, so check with the team if you can't sign up.

The Layout

The app has a sidebar on the left with everything organized under:

  • Dashboard — Quick overview with network stat cards and linked QuantAQ sensor performance alerts
  • Sensors — Full list of all sensors with search/filter
  • Audits — The audit pipeline (scheduled → in progress → finished, analysis pending → complete)
  • Collocations — The collocation pipeline (in progress → finished, analysis pending → complete)
  • Service Tickets — Track repairs and calibrations with QuantAQ, add progress notes, RMA and shipping numbers, etc.
  • Communities — All deployment communities organized by region, filterable by tags
  • Contacts — Community contacts grouped by location showing primary, active, and inactive statuses for each contact
Tip
You can pin frequently-used communities or sensors to the sidebar for quick access. Just click the star icon on any sensor or community detail page.

3. Working with Sensors

The All Sensors page is your main view. You can search by sensor ID, filter by status, and click any sensor to see its full detail page.

Sensor Statuses

Each sensor can have one or more status tags. Statuses fall into two groups: ones you apply manually, and ones that get applied automatically by workflows (collocations, audits, and Quant service tickets).

Manually applied statuses

You pick these from the status list on a sensor or in a Mass Action note.

  • Online — The sensor is deployed and reporting data. Mutually exclusive with Offline.
  • Offline — The sensor is not reporting data. Mutually exclusive with Online.
  • Lab Storage — The sensor is sitting at one of our lab locations, not currently deployed.
  • Ready for Deployment — The sensor has been checked out and is ready to ship or install.
  • Needs Repair — The sensor has a known problem and shouldn't be deployed until it's fixed.
  • PM Sensor Issue — The particulate matter sensor is malfunctioning.
  • Gaseous Sensor Issue — One or more of the gas sensors (NO2, O3, etc.) is malfunctioning.
  • SD Card Issue — The pod's SD card is malfunctioning.
  • Lost Connection — The sensor isn't reaching the network (covers power loss and connectivity drops).

Auto-applied statuses

These get added and removed automatically as you move sensors through the collocation tool, audit workflow, and Quant service tickets. You can still pick them from the status list, but the app will show a warning asking you to use the proper tool instead — the workflows keep everything in sync (history notes, dates, related records), and applying these by hand can leave the data out of step.

  • Collocation — Automatically applied when a sensor is part of an active collocation. Removed when the collocation completes. Use the Collocation tool to start one.
  • Auditing a Community — Automatically applied when an audit pod is actively auditing a community. Use the audit workflow (inside the Collocation tool) to start an audit.
  • In Transit Between Audits — Automatically applied to audit pods as they move between community audits. Managed by the audit workflow.
  • Service at Quant — Automatically applied by Quant service ticket progression when a sensor arrives at Quant. Advance the Quant ticket instead of setting this by hand.
  • Quant Ticket in Progress — Automatically applied whenever a Quant service ticket is open for the sensor. Removed when the ticket closes. Open a ticket from the Service section instead.

Moving a Sensor

When you need to move a single sensor to a different community, click "Move" on the sensor detail page. You can:

  • Select the destination community
  • Set the date of the move
  • Optionally change the sensor's status at the same time
  • Tag contacts who helped with the move
  • Add notes about the move

The app will ask if you want to update the install date (unless you're moving to a lab location — it skips the prompt for those since lab moves aren't installations).

Mass Action — Working with Multiple Sensors at Once

If you need to do something with several sensors at once — like installing 5 sensors at Garden after a collocation, or marking a batch of pods as Offline — use the Mass Action button at the top of the All Sensors page. It opens a note modal with bulk actions you can apply to all the sensors you tag.

Inside the Mass Action modal:

  • Move Sensors — Pick a destination community from the dropdown and every tagged sensor moves there in one click
  • Sensor Status Change — Pick statuses and they get added to every tagged sensor (existing tags like Collocation are preserved)
  • Troubleshooting / Site Work — Tag the action type for tracking
  • Tag the sensors, communities, and contacts involved
  • Write one note that gets attached to all of them
When to use Mass Action vs. the individual Move button
Use Mass Action when you're doing the same thing to multiple sensors at once — installing a batch at a site, taking a group offline for service, applying the same status change to several pods.

Use the individual Move button on a single sensor's detail page when you're handling just one sensor and want the install-date prompt and per-sensor context.

Sensor Detail Page

Each sensor has tabs for:

  • Event History — Every note, status change, movement, and service event. Filterable by category.
  • Collocations — All collocations this sensor was part of, plus its initial collocation from the original Salesforce data
  • Audits — Any audits involving this sensor
  • Service Tickets — Open and closed tickets with progress notes

4. Communities

Communities are organized alphabetically by parent communities, with child communities nested beneath them. You can change parent/child relationships anytime with the "Change Parent" button in each sensor's details page.

Each community page shows its sensors, contacts, event history, communications, and files. You can add sub-communities, deactivate communities you're no longer working with, and reactivate them later if needed.

How Community History Works
The community event history only shows notes that are explicitly tagged to that community. If a sensor moves to your community, only the movement note (and future notes tagged to your community) will show — not the sensor's entire history from previous communities; that will be visible under the sensor's individual history.

5. Notes & Event History

Notes are how everything gets documented in the app. When you add a note, you can:

  • Select Actions — Sensor Install, Sensor Removal, Troubleshooting, Site Work, and/or Sensor Status Change
  • If you check Sensor Status Change, you'll get a status picker and the status change gets merged into your note
  • Tag sensors, communities, and contacts so the note shows up in the right places
  • Use @mentions in the note text to tag contacts inline

The "Add Mass Note" button on the All Sensors page lets you create a note and tag multiple sensors/communities at once — handy for logging things like "Installed 3 sensors at NCore for collocation following their Quant calibration"

Editing Notes

Click the pencil icon on any note in the timeline to edit it. The edit modal shows the full text in a text area so you can make changes without losing any content. Click "Done" to save.

Event History Filters

On sensor pages, the event history dropdown lets you filter by:

  • Sensor Movement — Installs, removals, relocations
  • Troubleshooting / Site Work — Field work and issue resolution
  • Status Changes — Any status modifications
  • Communications — Logged emails, calls, site visits
  • General Notes — Everything else

6. Service Tickets

Service tickets track the lifecycle of a sensor being sent to QuantAQ for repair or calibration. Each ticket follows this workflow:

Ticket Opened
→
RMA Assigned
→
Shipped to Quant
→
At Quant
→
Shipped from Quant
→
Received

As you advance the ticket through each stage, the sensor's status updates automatically. For example, advancing to "Shipped to Quant" adds that status to the sensor and removes any previous shipping status.

Progress Notes

Each ticket has a Progress Notes section where you can add timestamped updates as things progress. These notes show on both the ticket detail and the sensor page's ticket preview, so you can see the latest status without clicking into the ticket.

Return Tracking

There are fields for return tracking info (to and from QuantAQ). Just enter the carrier and tracking number, like "UPS, 1234567890".

Closed Tickets

Closed tickets appear in their own section at the bottom of the Service Tickets page

7. Collocations

Collocations are when we place community sensors at a regulatory site (Garden, NCore, or Floyd Dryden) alongside a BAM reference monitor and/or permanent pod to verify the sensors are reading accurately.

Starting a Collocation

Click "+ Start Collocation" from the Collocations page in the sidebar. You'll pick:

  • Location — Regulatory sites are listed first, then all other communities alphabetically
  • Sensors — Tag the sensors being collocated
  • Conducted By — Who's running it
  • Notes — Any details

The sensors automatically get a "Collocation" status tag, and it's removed when the collocation completes.

Progress Notes

Just like service tickets, collocations have a progress notes section for tracking updates throughout the process. Add notes anytime from the collocation detail view.

8. Audits

Audits are similar to collocations but involve sending an audit pod to a community to run alongside the community's sensor for validation. The audit workflow goes: Scheduled → In Progress → Finished, Analysis Pending → Complete.

9. Data Analysis & Reports

This is the big one. Once a collocation or audit reaches "Complete," you can upload an Excel data file and the app runs the full regression analysis for you.

Uploading Data

  1. Export hourly data from AirVision for the BAM, permanent pod, and all community pods
  2. When selecting regulatory site data, use PM2.5 and PM10 local conditions
  3. Clean up any invalidated data and trim to the collocation dates
  4. Don't remove the first 24 hours — the app handles that automatically
  5. Define the BAM source (Garden BAM, NCore BAM, or Floyd Dryden BAM) and the permanent pod (optional; if no permanent pod is defined, all Quants will be compared against all other Quants instead of all Quants compared against the one permanent pod, all Quants get compared to the BAM data in both cases).
  6. Upload the Excel file

What You Get

The app automatically:

  • Parses all sensor columns from the spreadsheet
  • Excludes the first 24 hours from regression (the settling-in period)
  • Runs linear regression for each parameter (PM2.5, PM10, CO, NO, NO2, O3)
  • Checks DQO thresholds (R² ≥ 0.70, slope 0.65-1.35, intercept ±5)
  • Generates time series charts and regression scatter plots
  • Runs failsafe validation checks to verify the math is correct

Analysis Tabs

The analysis view has tabs matching our standard report format:

  • Time Series — Hourly plots for each parameter (tabbed by PM2.5, PM10, CO, etc.). Click sensors in the legend to toggle them off/on in the plot.
  • Pods vs BAM — Each sensor compared against the BAM (PM only)
  • Quants PM — Each pod compared against the permanent pod for PM, or each pod compared to each other pod if no permanent pod is defined
  • Quants Gaseous — Each pod compared against the permanent pod for gases (CO, NO, NO2, O3), or each pod compared to each other pod if no permanent pod is defined
  • Data Sheet — Full data table with yellow-shaded first 24 hours and red-highlighted missing/removed data

Saving Reports

Click "Save as HTML" to download a standalone report file. It opens in any browser with all the interactive charts — no server needed. The filename follows the format: Garden_Collocation_Analysis_445_451_674_Mar19_Apr1_2026.html

Validation Checks

Every analysis includes automated failsafe checks (click the validation dropdown to review):

  • Self-test — Verifies the regression engine is computing correctly using known test data
  • R² cross-check — Independently verifies each R² value using a different math method. The primary method uses least-squares regression and the cross-check uses Pearson correlation; if the two don't match, this is flagged in the validation dropdown on the analysis
  • Timestamp validation — Checks for data gaps, short datasets, and date issues
  • Sample size — Flags if there isn't enough data for reliable results
  • Borderline DQO — Alerts when a pass/fail result is very close to the threshold
Good to Know
X-PM sensors (the Modulair-PM models) only compare PM2.5 and PM10 — gas parameters are automatically skipped for these sensors since they don't measure gases.

10. Contacts

Contacts include anyone involved in the sensor network: tribal administrators, school staff, facility managers, etc. They're grouped by community in the Contacts view.

You can tag contacts in notes using @mentions (type @ and start typing their name), tag them in sensor movements, and add them to the email distribution list. Contacts can be deactivated (not deleted) when they're no longer involved, and reactivated later if needed.

11. Tips & Good-to-Knows

Timezones
Everything in the app runs on Alaska time (America/Anchorage).
Online & Offline
These two statuses are mutually exclusive everywhere in the app. If you select Online, Offline automatically deselects, and vice versa. No more accidentally having both.
Notes + Status Changes
When you add a note and also change the sensor status, they get combined into one entry in the event history — not two separate notes. The status change text gets appended to your note.
Community History
A community's event history only shows notes explicitly tagged to that community. If a sensor moves in, you won't see its old history from previous communities cluttering things up.
Re-uploading Analysis Data
You can re-upload data anytime from the collocation or audit detail view. Just click "Re-upload Data" and it'll clear the old results and let you start fresh. The old analysis is gone once you re-upload — there's no undo.
Deleting Things
When you delete an audit, collocation, or service ticket, all the auto-generated notes in event histories get cleaned up too. You don't need to manually delete those.
Setup Mode
Users with admin rights will see a "Setup Mode" toggle in the sidebar, that's for bulk data entry when first setting up the system. It shows inline edit fields on everything and skips creating history notes for changes. You probably don't need this for day-to-day use.
